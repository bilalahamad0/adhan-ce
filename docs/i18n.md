# Internationalization (i18n)

The extension UI can be switched to a different language **from inside the app**
(Settings → Language) and switches instantly, including right-to-left layout.

## Why a custom layer (not `chrome.i18n`)

`chrome.i18n.getMessage()` resolves against the **browser UI locale** and cannot be
changed at runtime — so it can't power an in-app language picker. We use a small
custom layer instead. (`chrome.i18n` is still the only way to localize the
installed name/description and the store listing — see "Deferred" below.)

## How it works

| Piece | Role |
|---|---|
| `lib/i18n.js` | The runtime layer. Loads JSON catalogs, persists the chosen language in `chrome.storage.local` (`lang`), exposes `t(key, params)` with English fallback. Imported by `popup.js` and `background.js`. |
| `locales/<code>.json` | Flat `key → string` catalogs. `en.json` is the **source of truth**; all others must match its key set. |
| `popup.html` | Static strings carry `data-i18n="key"` (textContent) or `data-i18n-attr="attr:key"` (attributes). Holds the `#lang` `<select>`. |
| `popup.js` | `initI18n()` → `applyDir()` → `applyStaticI18n()` before first render; dynamic strings go through `t()`; the `#lang` change handler persists + re-renders. |
| `content.js` | Classic content script — **can't import ES modules**, so it requests the catalog from the background via a `GET_I18N` message and uses a tiny inline `ti()`. Ships an inline English fallback so overlays never break. |
| `background.js` | Translates the desktop notification, and answers `GET_I18N` with `{ lang, dir, messages }` (English-merged catalog). |

Language fan-out is free: the choice lives in `chrome.storage.local`, and every
context already listens via `chrome.storage.onChanged` (`changes.lang`).

## Supported languages

`en` (source), `ar` ⏴RTL, `ur` ⏴RTL, `id`, `tr`, `fr`. Defined in `SUPPORTED`
(`lib/i18n.js`) and as `<option>`s in `popup.html`.

> ⚠️ **Translations except English are AI-assisted first passes and need native
> review** before you lean on them — especially Arabic/Urdu religious terminology
> and the focus-overlay copy. RTL layout bugs are easy to miss if you can't read
> the language; do a design pass with a native speaker.

## RTL handling

- `applyDir()` sets `<html dir/lang>`; the popup CSS uses logical properties
  (`margin-inline-start`, `inset-inline`) so it mirrors automatically.
- **Shadow DOM gotcha:** the content-script overlays use `:host { all: initial }`,
  which resets direction to LTR and does **not** inherit RTL across the boundary.
  So `content.js` sets `dir` explicitly on the overlay wrappers (`applyHostDir()`)
  and flips the corner card to bottom-**left** for RTL.

## Add a new language

1. Copy `locales/en.json` → `locales/<code>.json` and translate the **values**
   only. Keep every key and every `{placeholder}` exactly.
2. Add `<code>` to `SUPPORTED` in `lib/i18n.js`. If it's right-to-left, also add it
   to the `RTL` set.
3. Add an `<option value="<code>">Autonym</option>` to the `#lang` select in
   `popup.html` (label it in its own language, e.g. `العربية`).
4. `npm test` — the parity test (`tests/i18n.test.js`) fails if a key or a
   `{placeholder}` is missing/extra.
5. The packager already ships the whole `locales/` folder (`docs/pack.sh`).

## Deferred (browser-locale, not the in-app picker)

These are separate surfaces the runtime layer intentionally does **not** cover:

- **Installed name/description** → localize via `_locales/<code>/messages.json` +
  `default_locale` and `__MSG_…__` placeholders in `manifest.json`. This follows
  the **browser** locale (Chrome reads it at install/enable), so the in-app picker
  can't change it. Left out here to avoid extension-load risk until tested in a
  real browser.
- **Chrome Web Store listing** → localized per-language in the **Developer
  Dashboard**, gated by which `_locales/<code>` dirs you ship. Translate the
  detailed description + screenshots there.

## Tests

`tests/i18n.test.js` covers the pure helpers (`interpolate`, `makeT`,
`resolveLang`, `isRTLLang`) and enforces catalog integrity: every catalog has the
exact English key set and identical `{placeholders}`.
